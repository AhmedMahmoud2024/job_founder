Part 1: SOLID Principles in Flutter Development

1.1 Introduction to SOLID Principles

SOLID is an acronym for five design principles that make software more understandable, flexible, and maintainable. These principles were developed by Robert C. Martin (Uncle Bob) and have become industry standards for professional software development.

↪ SOLID Stands For:
  • Single Responsibility Principle
  • Open/Closed Principle
  • Liskov Substitution Principle
  • Interface Segregation Principle
  • Dependency Inversion Principle

Why SOLID Matters for Flutter Developers

  • Code Quality: Reduces bugs and improves reliability
  • Maintainability: Makes changes easier without breaking existing code
  • Testability: Enables unit testing through proper abstraction
  • Scalability: Allows app to grow without becoming unmaintainable
  • Team Collaboration: Clear architecture helps other developers understand code
  • Professional Standard: Industry best practice for production applications

1.2 Single Responsibility Principle (SRP)

Definition: A class should have one, and only one, reason to change. This means each class should have a single, well-defined responsibility.

Problem: Multiple Responsibilities

Before applying SRP, the HomeScreen had multiple responsibilities:

class HomeScreen extends StatelessWidget {
  const HomeScreen({super.key});

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      drawer: Drawer(...),  // ❌ Drawer responsibility
      body: Column(
        children: [
          // ❌ Drawer header logic
          DrawerHeader(...),
          
          // ❌ Welcome section logic
          Row(
            children: [
              Text('Welcome to Jobseek!'),
              Text('Discover Jobs'),
            ],
          ),
          
          // ❌ Search bar logic
          TextField(...),
          
          // ❌ Jobs list logic
          FeaturedJobsList(...),
          
          // ❌ Popular jobs logic
          PopularJobsList(...),
        ],
      ),
    );
  }
}

Problems:

  • 414 lines in a single file
  • 7 widget classes defined inside
  • Hard to test individual components
  • Any change in one section affects the entire file
  • Multiple reasons to change: drawer UI changes, search bar changes, jobs display changes, etc.

Solution: Extract into Separate Widgets

Apply SRP by creating separate, focused files:

lib/features/home/
├── home_screen.dart (77 lines)
└── widgets/
    ├── drawer_header.dart
    ├── welcome_header.dart
    ├── search_row.dart
    ├── featured_jobs_text.dart
    ├── featured_jobs_row.dart
    ├── popular_jobs_text.dart
    ├── first_popular_container.dart
    └── second_popular_container.dart

New HomeScreen (SRP Applied):

class HomeScreen extends StatelessWidget {
  const HomeScreen({super.key});

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      drawer: Drawer(
        child: SingleChildScrollView(
          child: Column(
            children: [
              DrowerHeaderOfHome(),
              // Drawer items...
            ],
          ),
        ),
      ),
      body: SafeArea(
        child: Padding(
          padding: const EdgeInsets.symmetric(horizontal: 24),
          child: SingleChildScrollView(
            child: Column(
              children: [
                WelcomeHeader(),
                SizedBox(height: 40),
                SearchRow(),
                SizedBox(height: 40),
                FeaturedJobsText(),
                SizedBox(height: 20),
                FeaturedJobsRow(),
                SizedBox(height: 40),
                PopularJobsText(),
                SizedBox(height: 17),
                FirstPopularContainer(),
                SizedBox(height: 20),
                SecondPopularContainer()
              ],
            ),
          ),
        ),
      ),
    );
  }
}

Each widget now has a single responsibility:

Widget Responsibility Lines of Code
DrowerHeaderOfHome Display user profile in drawer ~60
WelcomeHeader Display welcome greeting ~35
SearchRow Display search bar and filter ~40
FeaturedJobsRow Display featured jobs list ~50
FirstPopularContainer Display single job card ~40
✅ Benefits of SRP:
  • Each file ~50 lines instead of 414 lines
  • Easy to test individual widgets
  • Easy to reuse widgets in other screens
  • Single reason to change each widget
  • Better readability and maintainability

1.3 Open/Closed Principle (OCP)

Definition: Classes should be open for extension but closed for modification. You should be able to add new functionality without changing existing code.

Problem: Code Duplication

Before OCP, every repository duplicated error handling:

class JobsRepo {
  final ApiServices apiServices;
  
  Future<ApiResult<JobsDataModel>> getJobs() async {
    try {
      final data = await apiServices.getJobs();
      return ApiResult.success(data);
    } catch(e) {
      return ApiResult.failure(e.toString());
    }
  }
}

class LoginRepo {
  final ApiServices apiServices;
  
  Future<ApiResult<LoginModel>> login(...) async {
    try {
      final response = await apiServices.login(...);
      return ApiResult.success(response);
    } catch(e) {
      return ApiResult.failure(e.toString());
    }
  }
}

class RegisterRepo {
  final ApiServices apiServices;
  
  Future<ApiResult<RegisterModel>> register(...) async {
    try {
      final response = await apiServices.register(...);
      return ApiResult.success(response);
    } catch(e) {
      return ApiResult.failure(e.toString());
    }
  }
}

Problems:

  • Error handling code duplicated 3+ times
  • 6 lines of try-catch per repository
  • Bug fix requires changes in multiple places
  • New repositories must copy-paste the pattern
  • Violations of DRY principle (Don't Repeat Yourself)

Solution: Create Base Class with Error Handling

Step 1: Create BaseRepository

abstract class BaseRepository {
  Future<ApiResult<T>> handleApiCall<T>(
    Future<T> Function() apiCall,
  ) async {
    try {
      final result = await apiCall();
      return ApiResult.success(result);
    } catch(e) {
      return ApiResult.failure(e.toString());
    }
  }
}

Step 2: Repos Extend BaseRepository

class JobsRepo extends BaseRepository {
  final ApiServices apiServices;

  JobsRepo({required this.apiServices});

  @override
  Future<ApiResult<JobsDataModel>> getJobs() async =>
      handleApiCall(() => apiServices.getJobs());
}

class LoginRepo extends BaseRepository {
  final ApiServices apiServices;

  LoginRepo({required this.apiServices});

  @override
  Future<ApiResult<LoginModel>> login(...) async =>
      handleApiCall(() => apiServices.login(...));
}

❌ Before OCP

  • 6 lines error handling per repo
  • 3 repos × 6 lines = 18 lines duplicated
  • New repo: copy-paste again
  • Bug fix: change 3+ places

✅ After OCP

  • Error handling in BaseRepository once
  • 1 line per repo method
  • New repo: extend + 1 line
  • Bug fix: change BaseRepository only
✅ Benefits of OCP:
  • Error handling code written once, used everywhere
  • Adding new repositories is trivial
  • Bug fixes apply globally
  • DRY principle enforced
  • Code is more maintainable and scalable

1.6 Dependency Inversion Principle (DIP)

Definition: High-level modules should not depend on low-level modules. Both should depend on abstractions. Depend on interfaces, not concrete implementations.

Problem: Tight Coupling

Before DIP: Cubits depend on concrete repository classes

class JobsCubit extends Cubit<JobsState> {
  final JobsRepo jobsRepo;  // ❌ Concrete class, not interface
  
  JobsCubit(this.jobsRepo) : super(JobsState.initial());
  
  Future<void> getJobs() async {
    emit(JobsState.loading());
    final response = await jobsRepo.getJobs();
    // ...
  }
}

class LoginCubit extends Cubit<LoginState> {
  final LoginRepo loginRepo;  // ❌ Concrete class, not interface
  
  LoginCubit(this.loginRepo) : super(LoginState.initial());
}

Problems:

  • Cubits tightly coupled to concrete repository implementations
  • Can't swap real repo with mock repo for testing
  • Must create entire JobsRepo just to test JobsCubit
  • Hard to extend without modifying cubit code
  • Tests become slow and dependent on APIs

Solution: Use Interfaces (Abstract Classes)

Step 1: Create Repository Interfaces

abstract class IJobsRepository {
  Future<ApiResult<JobsDataModel>> getJobs();
}

abstract class IAuthRepository {
  Future<ApiResult<LoginModel>> login({
    required String email,
    required String password,
  });

  Future<ApiResult<RegisterModel>> register({
    required String email,
    required String password,
  });
}

Step 2: Implementations Follow Interface

class JobsRepo extends BaseRepository implements IJobsRepository {
  final ApiServices apiServices;

  JobsRepo({required this.apiServices});

  @override
  Future<ApiResult<JobsDataModel>> getJobs() async =>
      handleApiCall(() => apiServices.getJobs());
}

class LoginRepo extends BaseRepository implements IAuthRepository {
  final ApiServices apiServices;

  LoginRepo({required this.apiServices});

  @override
  Future<ApiResult<LoginModel>> login(...) async =>
      handleApiCall(() => apiServices.login(...));
  
  @override
  Future<ApiResult<RegisterModel>> register(...) async =>
      handleApiCall(() => apiServices.register(...));
}

Step 3: Cubits Depend on Interfaces

class JobsCubit extends Cubit<JobsState> {
  final IJobsRepository jobsRepository;  // ✅ Interface, not concrete
  
  JobsCubit(this.jobsRepository) : super(JobsState.initial());
  
  Future<void> getJobs() async {
    emit(JobsState.loading());
    final response = await jobsRepository.getJobs();
    response.when(
      success: (data) => emit(JobsState.success(data)),
      failure: (error) => emit(JobsState.fail(error)),
    );
  }
}

class LoginCubit extends Cubit<LoginState> {
  final IAuthRepository authRepository;  // ✅ Interface, not concrete
  
  LoginCubit(this.authRepository) : super(LoginState.initial());
  
  Future<void> login({required String email, required String password}) async {
    emit(LoginState.loading());
    final response = await authRepository.login(email: email, password: password);
    response.when(
      success: (data) => emit(LoginState.success(data)),
      failure: (error) => emit(LoginState.fail(error)),
    );
  }
}

Benefits for Testing

With DIP, you can easily mock repositories:

class MockJobsRepository implements IJobsRepository {
  @override
  Future<ApiResult<JobsDataModel>> getJobs() async {
    return ApiResult.success(
      JobsDataModel(
        jobs: [
          JobsModel(id: 1, jobTitle: 'Flutter Dev', companyName: 'Tech Corp'),
          JobsModel(id: 2, jobTitle: 'Web Dev', companyName: 'Web Inc'),
        ],
      ),
    );
  }
}

void main() {
  test('JobsCubit emits success state when getJobs succeeds', () async {
    final mockRepo = MockJobsRepository();
    final cubit = JobsCubit(mockRepo);
    
    await cubit.getJobs();
    
    expect(cubit.state, isA<JobsStateSuccess>());
    verify(mockRepo.getJobs).called(1);
  });
}
✅ Benefits of DIP:
  • Easy to test without real API calls
  • Swap implementations without changing cubit
  • Loose coupling between layers
  • Extensible: add new repo types easily
  • Better separation of concerns

Part 2: BLoC/Cubit State Management

2.1 What is BLoC/Cubit?

BLoC stands for Business Logic Component. It's a design pattern for managing state in Flutter applications.

BLoC vs Cubit

Feature BLoC Cubit
Complexity More complex, event-driven Simpler, method-based
Method Call bloc.add(Event) cubit.method()
Boilerplate Event classes + handlers Just methods
Event History Clear event stream No event history
Best For Complex state logic Simple to medium apps
Your App Not used ✅ Used for state management
ℹ️ Your Choice: Cubit

Your Job Founder app uses Cubit because it's simpler and more straightforward for managing state without unnecessary complexity. Cubit is perfect for apps that don't need complex event-driven architecture.

2.2 Cubit Pattern in Detail

The Three Components

  1. State (Immutable Data): Represents the current state of the Cubit
  2. Cubit (Business Logic): Contains methods that emit new states
  3. BlocBuilder (UI Listener): Watches Cubit and rebuilds UI when state changes

1. Define Immutable States (using Freezed)

import 'package:freezed_annotation/freezed_annotation.dart';

part 'jobs_state.freezed.dart';

@freezed
class JobsState with _$JobsState {
  const factory JobsState.initial() = _Initial;
  
  const factory JobsState.loading() = _Loading;
  
  const factory JobsState.success(JobsDataModel data) = _Success;
  
  const factory JobsState.fail(String error) = _Fail;
}

What Freezed Provides:

  • Immutable instances (can't be accidentally modified)
  • Sealed unions (each state is a distinct type)
  • Pattern matching with when() method
  • Automatic copyWith() for cloning
  • Proper == operator and toString()

2. Create Cubit with Business Logic

import 'package:flutter_bloc/flutter_bloc.dart';

class JobsCubit extends Cubit<JobsState> {
  final IJobsRepository jobsRepository;
  List<Job> jobs = [];
  
  JobsCubit(this.jobsRepository) : super(JobsState.initial());
  
  Future<void> getJobs() async {
    emit(JobsState.loading());
    try {
      final response = await jobsRepository.getJobs();
      response.when(
        success: (data) {
          jobs = data.jobs;
          emit(JobsState.success(data));
        },
        failure: (error) => emit(JobsState.fail(error)),
      );
    } catch(e) {
      emit(JobsState.fail(e.toString()));
    }
  }
}

Key Methods:

  • emit(state) - Notify listeners of state change
  • jobsRepository.getJobs() - Call data layer
  • response.when(...) - Pattern match API result

3. Register in Dependency Injection

void _setupJobsDependencies() {
  // Repository as singleton (shared instance)
  getIt.registerLazySingleton<IJobsRepository>(
    () => JobsRepository(apiServices: getIt<ApiServices>()),
  );
  
  // Cubit as factory (new instance per widget)
  getIt.registerFactory<JobsCubit>(
    () => JobsCubit(getIt<IJobsRepository>()),
  );
}

Registration Strategy:

  • registerLazySingleton for repositories (one instance, shared)
  • registerFactory for cubits (new instance per widget)

4. Use in Widget with BlocBuilder

class FeaturedJobsRow extends StatelessWidget {
  const FeaturedJobsRow({super.key});

  @override
  Widget build(BuildContext context) {
    return BlocBuilder<JobsCubit, JobsState>(
      builder: (context, state) => state.when(
        initial: () => const SizedBox(),
        loading: () => const Center(
          child: CircularProgressIndicator(),
        ),
        success: (data) => SingleChildScrollView(
          scrollDirection: Axis.horizontal,
          child: Row(
            children: [
              ...List.generate(data.jobs.length, (index) {
                return JobCard(job: data.jobs[index]);
              }),
            ],
          ),
        ),
        fail: (error) => Center(
          child: Text('Error: $error'),
        ),
      ),
    );
  }
}

Pattern Matching with when():

  • Type-safe: compiler checks all states covered
  • No casting needed
  • Each state handled separately
  • Impossible states prevented

5. Trigger from UI

onPressed: () {
  context.read<JobsCubit>().getJobs();
}

Complete Flow Diagram (Text Description)

        ↓

        ↓

        ↓
emit(JobsState.loading())
        ↓
BlocBuilder receives state change
        ↓
UI rebuilds with loading spinner
        ↓
jobsRepository.getJobs() called
        ↓
IJobsRepository interface routed
        ↓
JobsRepository implementation executes
        ↓
BaseRepository.handleApiCall() wraps with error handling
        ↓
ApiServices.getJobs() makes HTTP request
        ↓
API returns data or throws error
        ↓
BaseRepository wraps in ApiResult
        ↓
Result returned to JobsCubit
        ↓
emit(JobsState.success(data)) or emit(JobsState.fail(error))
        ↓
BlocBuilder receives new state
        ↓
UI rebuilds with data or error message

Architecture Layers

Layer Components Responsibility
Presentation Widgets, BlocBuilder Display UI, trigger user actions
Business Logic Cubit, Methods Manage state, coordinate between UI and data
Domain Interfaces, Contracts Define what operations are available
Data Repository, Models Fetch and manage data
Network ApiServices, Dio HTTP requests to remote server

2.5 Testing Strategies

Unit Testing Cubit

import 'package:bloc_test/bloc_test.dart';
import 'package:mockito/mockito.dart';

class MockJobsRepository extends Mock implements IJobsRepository {}

void main() {
  group('JobsCubit Tests', () {
    late JobsCubit jobsCubit;
    late MockJobsRepository mockRepository;
    
    setUp(() {
      mockRepository = MockJobsRepository();
      jobsCubit = JobsCubit(mockRepository);
    });
    
    tearDown(() {
      jobsCubit.close();
    });
    
    test('initial state is JobsState.initial()', () {
      expect(jobsCubit.state, isA<Initial>());
    });
    
    blocTest<JobsCubit, JobsState>(
      'emits [Loading, Success] when getJobs succeeds',
      build: () {
        when(mockRepository.getJobs()).thenAnswer(
          (_) async => ApiResult.success(
            JobsDataModel(jobs: [/* mock jobs */]),
          ),
        );
        return jobsCubit;
      },
      act: (cubit) => cubit.getJobs(),
      expect: () => [
        isA<Loading>(),
        isA<Success>(),
      ],
    );
    
    blocTest<JobsCubit, JobsState>(
      'emits [Loading, Failure] when getJobs fails',
      build: () {
        when(mockRepository.getJobs()).thenAnswer(
          (_) async => ApiResult.failure('Network error'),
        );
        return jobsCubit;
      },
      act: (cubit) => cubit.getJobs(),
      expect: () => [
        isA<Loading>(),
        isA<Fail>(),
      ],
    );
  });
}

Widget Testing

void main() {
  testWidgets('FeaturedJobsRow displays jobs', (WidgetTester tester) async {
    await tester.pumpWidget(
      MaterialApp(
        home: BlocProvider(
          create: (context) => JobsCubit(mockRepository)..getJobs(),
          child: Scaffold(
            body: FeaturedJobsRow(),
          ),
        ),
      ),
    );
    
    // Wait for loading state
    await tester.pumpAndSettle();
    
    // Verify job cards are displayed
    expect(find.byType(JobCard), findsWidgets);
  });
}

Different Test Levels

  • Widget Test: Test UI rebuild with real cubit
